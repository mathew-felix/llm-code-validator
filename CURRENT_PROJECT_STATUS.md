# AI Code Hallucination Validator — Current Project Status

This document reflects the project as it exists right now in the repository, not the earlier design target.

## What Is Done

The project is built as a working end-to-end validator for AI-generated Python code. A user can paste code into the frontend, send it to the FastAPI backend, run it through the LangGraph workflow, and receive a structured validation report with issue cards, corrected code suggestions, confidence scores, and a plain-English summary.

The major project pieces are already implemented:

- `frontend/index.html`: demo UI for pasting code and rendering validation results
- `api/main.py`: FastAPI server with `POST /validate`
- `agent/graph.py`: compiled LangGraph workflow
- `agent/nodes/*.py`: AST extraction, database lookup, PyPI fallback, supervisor routing, specialist checks, LLM diagnosis, and report generation
- `agent/schemas.py`: Pydantic models for issues, reports, and state
- `data/library_signatures.json`: curated breaking-change database
- `tests/evaluate.py`: benchmark runner over the 50-case dataset
- `tests/baseline.py`: exact-match baseline for comparison
- `thesis_validation/run_on_thesis.py`: real-world repo scan against the thesis project

In practical terms, this is no longer just a plan or scaffold. It is a runnable portfolio project with a live API path, a demo UI, benchmark scripts, and a real-world test harness.

## Current Architecture

The current graph is more complex than the original five-node pipeline. Conceptually, the flow is:

1. `extract_imports`
2. `check_database`
3. Optional `fetch_pypi` if unknown libraries appear
4. `supervisor`
5. Optional `import_specialist`
6. Optional `method_specialist`
7. `llm_diagnose`
8. `generate_report`
9. `END`

Important details about the implemented architecture:

- The AST node statically extracts imports and attribute calls without executing user code.
- The database node matches extracted calls against the curated JSON signature store.
- PyPI fallback runs only when a library is outside the local database.
- A supervisor node decides whether the import specialist, method specialist, or both should run.
- Specialist outputs are merged with the broader LLM diagnosis pass rather than replacing it.
- The report node deduplicates overlapping findings and filters out issues below the confidence threshold of `0.75`.

So the project currently uses both deterministic routing and LLM-driven routing:

- Deterministic routing for the PyPI skip/fetch decision
- LLM-driven routing for specialist selection

## What The Project Is Currently Capable Of

The validator is currently capable of the following:

- Detecting many known deprecated imports, removed methods, and migration-related API breakages across 20 curated libraries
- Returning typed JSON output with issue type, line number, explanation, corrected line, corrected full code, overall confidence, and filtered issue count
- Handling code snippets through the live `/validate` API endpoint
- Running an automated evaluation benchmark over the validation dataset
- Running a repo scan over another Python project through the thesis-validation script
- Falling back to deterministic database and PyPI evidence when the LLM layer times out or quota is unavailable
- Producing richer explanations than a simple string-matching baseline because the LLM reasons over database matches and PyPI evidence

The project is best described as an API breakage validator for AI-generated Python snippets, not as a general-purpose linter, type checker, or full code intelligence system.

## Measured Capability

The current benchmark numbers come from `validation_dataset/results.json`, generated by `tests/evaluate.py` over 50 labeled test cases containing 82 known issues.

Current full-agent benchmark:

- Precision: `76.6%`
- Recall: `72.0%`
- True positives: `59`
- False positives: `18`
- False negatives: `23`
- Average runtime per test: `7.03s`

Baseline benchmark from `tests/baseline.py`:

- Precision: `86.7%`
- Recall: `80.2%`

What this means in plain English:

- The full agent works and catches many real migration and deprecation errors.
- It produces better explanations and richer output than a pure dictionary lookup.
- But on the current benchmark, it does **not** outperform the simpler exact-match baseline in raw precision or recall.

That is the most honest current assessment of the system. The project is useful and demonstrable, but its measured accuracy is still behind the simpler baseline on this dataset.

## Where It Is Strong

In the current benchmark, the project performs best when:

- The breakage is already represented clearly in the curated database
- The code uses explicit import paths or explicit method names
- The validation problem is an API migration pattern rather than a type-inference problem

It is currently strongest on database-driven import and migration breakages such as:

- LangChain import-path changes
- LlamaIndex migration issues
- NumPy alias removals
- Pinecone API migration patterns
- ChromaDB client migration cases
- Several Torch and LangGraph deprecations

## Problems The Project Faced

The project currently faces five major problems.

### 1. The full agent still trails the simple baseline

This is the biggest engineering reality in the repo today. The project added supervisor routing, specialists, confidence filtering, and a broader diagnosis stage, but that extra complexity did not produce a better benchmark score than the exact-match baseline.

So the architecture is more advanced, but the measured accuracy still needs improvement.

### 2. Method attribution is still hard

Calls like `df.append()` are harder to attribute reliably than explicit imports because the validator does not do full type inference. This hurts recall on some libraries and is one of the reasons the exact-match baseline still competes strongly.

### 3. Import-name normalization is still heuristic

The validator now normalizes common import-name vs distribution-name mismatches before the PyPI lookup. That fixed the thesis false positives for `yaml` (`PyYAML`) and `cv2` (`opencv-python`). But the approach is still a curated lookup table, so unusual import/distribution mismatches can still slip through until they are added explicitly.

### 4. The current product is optimized for snippets, not large files

The API path currently enforces a 10,000-character input limit. That is reasonable for pasted LLM snippets, but it becomes a real constraint for repo-scale validation because larger source files are skipped in the thesis scan.

### 5. LLM availability can still narrow the result quality

The project now has a deterministic fallback when OpenAI times out or returns quota errors, so clearly provable issues still surface instead of failing open. But when that fallback path is active, the system loses the richer explanation and broader reasoning quality that the full LLM diagnosis pass provides.

## What Happened When Testing On The Thesis Repo

The repo now includes `neural-edge-video-compression`, which was used as the real-world test project.

The thesis-validation harness in `thesis_validation/run_on_thesis.py` currently:

- walks the thesis repo
- skips vendored and generated directories such as `DCVC`, `_third_party_amt`, `tests`, `venv`, and cache folders
- skips files above the current 10,000-character validator limit
- runs the validator on the remaining project-owned Python files
- writes flagged files to `thesis_validation/results.json`

Current thesis scan coverage:

- Project-owned Python files after directory filtering: `28`
- Files within the validator's size limit: `15`
- Files skipped for exceeding the size limit: `13`

Current thesis scan outcome:

- Files flagged by the validator: `1`

Examples from the thesis run:

- `src/decompression/interpolation_amt.py`

What those findings actually meant:

- The previous local/relative import false positives were removed by the local-module filter
- The import-name vs distribution-name false positives were removed by the normalization table:
  `yaml` now maps to `PyYAML`, and `cv2` now maps to `opencv-python`
- The remaining flagged file, `src/decompression/interpolation_amt.py`, contains a plausible real deprecation on `torch.autocast`

So the thesis test is no longer dominated by false positives. After the local-module filter and import-name normalization pass, the scan now collapses to a single plausible real migration issue instead of multiple obvious mistakes. That is a much stronger real-world result than the earlier runs.

## Honest Summary Of Current Capability

Right now the project is:

- complete enough to demo
- complete enough to benchmark
- complete enough to discuss as an agentic validation system
- complete enough to connect credibly to the thesis project

Right now the project is **not**:

- better than the simple baseline on benchmark accuracy
- fully reliable for repo-wide validation of local-module-heavy codebases
- a replacement for linting, typing, or full static analysis

## Bottom Line

The project is currently a solid portfolio-grade AI engineering system with:

- a real API
- a real LangGraph workflow
- structured outputs
- automated evaluation
- a thesis-repo integration test

Its strongest value today is as a demonstrable AI tooling project that validates LLM-generated Python against curated API knowledge and conditional agent routing.

Its biggest current weaknesses are:

- lower benchmark accuracy than the simple baseline
- heuristic handling for repo-local modules and uncommon import/distribution mismatches
- dependence on the LLM layer for the richest explanations and corrections
- limited support for large-file and repo-scale validation

If someone asks, "What is the current state of the project?" the most accurate answer is:

The project works, is fully runnable, and demonstrates a real AI-agent validation pipeline. It now gets the thesis scan down to one plausible real issue, but it still needs another iteration on method attribution and large-file support before it can claim strong repo-scale reliability.
